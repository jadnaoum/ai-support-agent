# Eval workflow

## Running evals

```bash
# Full baseline
python -m evals.run_evals --tag "baseline_v1" --delay 5 --yes \
  --desc "Description of what changed"

# Single sheet
python -m evals.run_evals --tag "v1.1_fix" --sheets "Intent Classifier" --delay 5 --yes

# Judge-only re-run (re-judge a prior run's responses, no agent calls)
python -m evals.run_evals --tag "cal_v1" --judge-only --from-tag "baseline_v1" \
  --judge-model opus --delay 5 --yes \
  --desc "Opus calibration re-judge"
# Outputs: {tag}_disagreements.txt listing every case where verdicts differ
```

Key flags:
- `--tag` — required; used as column header in xlsx and Run History
- `--delay` — seconds between test cases (default 5). Do not set below 2 without checking rate limits.
- `--yes` — skips cost confirmation; safe for scripted/CI runs
- `--calibrate` — uses Opus for all judge calls instead of tiered Haiku/Sonnet
- `--judge-only` — skip agent; re-judge stored responses from `--from-tag` run. Requires sidecar to exist (auto-generated by full runs). Skips Input Guard, Intent Classifier, Output Guard (programmatic).
- `--from-tag` — baseline tag to read responses from (used with `--judge-only`; auto-detects most recent if omitted)
- `--judge-model opus` — same as `--calibrate`

## Rate limit math (Tier 1)

Anthropic Tier 1 = 50 RPM per model. Each eval case makes roughly:
- 1 agent call → 2–3 LLM calls (input guard + intent + response) against Sonnet
- 1 judge call → 1 LLM call against Sonnet (behavioral/safety) or Haiku (classification)

Peak throughput at `--delay 5`: ~12 cases/min × ~4 calls/case = ~48 RPM. This fits within 50 RPM with a small margin for retries. If you see `rate_limit` in failure_reason columns, increase `--delay`.

## Judge architecture

| Sheet                 | Judge type     | Model  |
|-----------------------|---------------|--------|
| Input Guard           | programmatic   | —      |
| Intent Classifier     | programmatic   | —      |
| Output Guard          | programmatic + LLM | Haiku |
| KB Retrieval          | LLM-as-judge   | Sonnet |
| Action Execution      | LLM-as-judge   | Sonnet |
| Escalation            | LLM-as-judge   | Sonnet |
| Conversation Quality  | LLM-as-judge   | Sonnet |
| PII & Data Leakage    | LLM-as-judge   | Sonnet |
| Policy Compliance     | LLM-as-judge   | Sonnet |
| Graceful Failure      | LLM-as-judge   | Sonnet |
| Context Retention     | LLM-as-judge   | Sonnet |

## Interpreting results

- `rate_limit` in failure_reason → infrastructure failure, not agent behavior. Increase `--delay` and re-run that sheet.
- `api_timeout` in failure_reason → single call hung past 90s. Transient; re-run.
- `judge_error` → LLM judge returned unparseable JSON or hard error. Check the reasoning column.

## File locations

- Test cases + results: `evals/eval_test_cases.xlsx`
- Agent response sidecars: `evals/eval_runs/{tag}/responses.json`
- Judge rubrics: `prompts/eval_rubrics.yaml`
- Runner: `evals/run_evals.py`
- Judge modules: `evals/judges/{classification,behavioral,safety}.py`
- Run logs: `baseline_v1.log` (or equivalent tag name)

## Sidecar format

Every full run saves `evals/eval_runs/{tag}/responses.json` — a JSON object keyed by test_id:

```json
{
  "KB-001": {
    "response": "full untruncated agent response text",
    "actions_taken": [{"service": "knowledge", ...}],
    "requires_escalation": false,
    "escalation_reason": "",
    "confidence": 0.92,
    "inferred_intent": "knowledge_query",
    "context_summary": ""
  }
}
```

This is the ground truth for `--judge-only` reruns. The xlsx only stores 500-char truncated response text and no tool-call data. If the sidecar is missing for a `--from-tag`, the runner errors immediately rather than silently using empty `actions_taken`.
